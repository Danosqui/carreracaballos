Cambio 1 — persistence.xml

src/main/resources/META-INF/persistence.xml Configuración
Problema A: El nombre de la unidad de persistencia era herenciaPU (copiado de otro ejercicio), pero JPAUtil.java busca carreracaballosPU. JPA lanza No Persistence provider for EntityManager named carreracaballosPU.
persistence.xml — nombre de unidad
<persistence-unit name="herenciaPU" transaction-type="RESOURCE_LOCAL"> <persistence-unit name="carreracaballosPU" transaction-type="RESOURCE_LOCAL">
Problema B: La contraseña root era el valor de ejemplo. MySQL devolvía Access denied (error 1045).
persistence.xml — contraseña
<property name="jakarta.persistence.jdbc.password" value="root"/> <property name="jakarta.persistence.jdbc.password" value="admin123"/>
Error que producía: La aplicación no podía iniciar — fallaba al crear el EntityManagerFactory.

Cambio 3 — Caballo.java

src/.../model/Caballo.java Bug
Problema: JPA/Hibernate requiere un constructor sin argumentos en toda entidad para poder instanciar objetos al leer desde la base de datos. Sin él, las subclases tampoco podían declarar su propio constructor vacío (el compilador busca implícitamente super()).
Caballo.java — constructor vacío
public abstract class Caballo { protected Caballo() {} public Caballo(String nombre) { ... }
Error que producía: Error de compilación en las tres subclases + Hibernate no podía reconstruir objetos al hacer SELECT.

Cambio 4 — Jugador.java

src/.../model/Jugador.java Bug
Problema: Mismo problema que Caballo — JPA no puede instanciar la entidad sin constructor vacío. Hibernate lo advertía en los logs: No default (no-argument) constructor for class Jugador.
Jugador.java — constructor vacío
public class Jugador { protected Jugador() {} public Jugador(String nombre, String mail, int puntaje) { ... }
Error que producía: Al hacer listarJugadores() o buscarJugador(), Hibernate no podía crear los objetos de retorno.

Cambios 5, 6 y 7 — CaballoVeloz / CaballoResistente / CaballoEquilibrado

src/.../model/CaballoVeloz.java  |  CaballoResistente.java  |  CaballoEquilibrado.java Arquitectura
Problema A — anotaciones duplicadas en subclases SINGLE_TABLE: Con la estrategia SINGLE_TABLE, SOLO la clase raíz (Caballo) define la tabla y la estrategia de herencia. Las subclases NO deben llevar @Table, @Inheritance ni @DiscriminatorColumn. Hibernate lanzaba: Entity is a subclass in a SINGLE_TABLE hierarchy and may not be annotated @Table.
Problema B — nombre de tabla inconsistente: Caballo usaba @Table(name="caballo") y las subclases @Table(name="caballos") — singular vs plural, dos tablas distintas.
Problema C — falta constructor vacío: Igual que en Caballo.java.
CaballoVeloz.java (igual para Resistente y Equilibrado)
import jakarta.persistence.DiscriminatorColumn; import jakarta.persistence.Inheritance; import jakarta.persistence.InheritanceType; import jakarta.persistence.Table; import jakarta.persistence.DiscriminatorValue; @Entity @Table(name="caballos") @Inheritance(strategy=InheritanceType.SINGLE_TABLE) @DiscriminatorColumn(name="tipo_caballo") public class CaballoVeloz extends Caballo { @Entity @DiscriminatorValue("VELOZ") public class CaballoVeloz extends Caballo { protected CaballoVeloz() {} public CaballoVeloz(String nombre) { ... }
Regla: en herencia SINGLE_TABLE, la tabla la define la clase raíz. Las subclases solo agregan @Entity y @DiscriminatorValue("VALOR") para identificar su fila en la columna discriminadora.
Error que producía: AnnotationException al iniciar — Hibernate rechazaba el modelo antes de ejecutar cualquier query.

Cambio 8 — CaballoDAO.java

src/.../dao/CaballoDAO.java Bug
Problema A — JPQL usa nombre de tabla en vez de nombre de clase: JPQL (Java Persistence Query Language) opera sobre las clases Java, no sobre las tablas SQL. El nombre debe coincidir exactamente con el nombre de la clase anotada con @Entity.
CaballoDAO.java — listarCaballos()
return em.createQuery("SELECT c FROM caballo c", Caballo.class) return em.createQuery("SELECT c FROM Caballo c", Caballo.class)
CaballoDAO.java — getRandomCaballos()
return em.createQuery("SELECT c FROM caballo c WHERE c.id <> :excluir", Caballo.class) return em.createQuery("SELECT c FROM Caballo c WHERE c.id <> :excluir", Caballo.class)
Problema B — remove sobre entidad detached: Cuando se recibe un objeto de fuera del EntityManager actual, ese objeto está en estado "detached" (desconectado). Llamar remove() sobre él lanza una excepción. Primero hay que "reattacharlo" con merge().
CaballoDAO.java — borrarCaballo()
em.remove(caballo); em.remove(em.merge(caballo));
Error que producían: A — UnknownEntityException: Could not resolve root entity 'caballo' al listar. B — IllegalArgumentException: Removing a detached instance al borrar.

Cambio 9 — JugadorDAO.java

src/.../dao/JugadorDAO.java Bug
Problema A — JPQL con nombre de tabla: Mismo problema que CaballoDAO.
JugadorDAO.java — listarJugadores()
return em.createQuery("SELECT j FROM jugador j", Jugador.class) return em.createQuery("SELECT j FROM Jugador j", Jugador.class)
Problema B — excepción silenciosa en crearJugador(): El bloque catch hacía rollback pero no relanzaba la excepción. Si el INSERT fallaba, el código que llamó al método nunca lo sabía — falla silenciosa.
JugadorDAO.java — crearJugador()
catch(RuntimeException e) { if (em.getTransaction().isActive()) em.getTransaction().rollback(); throw e; // relanzar para que el error sea visible }
Problema C — getSingleResult() frágil: getSingleResult() lanza NonUniqueResultException si hay más de un jugador con el mismo nombre. Ocurrió en las pruebas con registros duplicados de ejecuciones anteriores.
JugadorDAO.java — buscarJugador()
return enti.createQuery(...).setParameter("nombre", nombre).getSingleResult(); List<Jugador> resultados = enti.createQuery(...) .setParameter("nombre", nombre).setMaxResults(1).getResultList(); return resultados.isEmpty() ? null : resultados.get(0);
Problema D — remove sobre entidad detached: Mismo problema que CaballoDAO.
JugadorDAO.java — borrarJugador()
em.remove(jugador); em.remove(em.merge(jugador));
Errores que producían: A — UnknownEntityException al listar. B — falla silenciosa al crear. C — NonUniqueResultException con datos duplicados. D — IllegalArgumentException al borrar.

Cambio 10 — DAOTest.java

private static void limpiarDB() { EntityManager em = JPAUtil.getInstance().crearEntityManager(); try { em.getTransaction().begin(); em.createQuery("DELETE FROM Jugador").executeUpdate(); em.createQuery("DELETE FROM Caballo").executeUpdate(); em.getTransaction().commit(); } finally { em.close(); } }

try { limpiarDB(); // siempre partir de DB vacía testCaballoDAO(); testJugadorDAO();
Efecto: El test es ahora idempotente — se puede ejecutar N veces seguidas con el mismo resultado.
